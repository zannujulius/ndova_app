Remote repositories live on GitHub. Push your commits and share work with your team. Pull requests are how you propose merging your branch into main. Every PR must be reviewed before merging.

main — production-ready code only. development — active integration branch. feature/xxx — one branch per task. hotfix/xxx — urgent production fixes. release/xxx — preparing for a release. # **Lesson 1 Introduction to TypeScript** 

May 2026 

**What Is TypeScript? JS With a Type System** TypeScript is JavaScript with static types. It catches errors before runtime — at the point you write the code. Your editor highlights problems instantly as you type. Type inference: TypeScript figures out types from context. Explicit annotation: you declare what type something is. Works seamlessly with both React frontend and Node.js backend. 

May 2026 

## **Lesson 2 Basic Types & Annotations** 

## **The Everyday Types** 

string, number, boolean — the three primitives. string[] or number[] — typed arrays. null and undefined — be explicit about absence. 

Function types: function getStatus(id: number): string Return types prevent unexpected undefined values in API handlers and component functions. 

May 2026 

# **Interfaces & Type Aliases** 

**interface type alias** 

Defines the shape of an object. Add ? for optional properties. Use readonly to prevent mutations. Ideal for data models: User, Service, and Appointment. Similar to interface but more flexible. Can represent unions, tuples, and primitives too. Use type for complex compositions, interface for object shapes you intend to extend. 

May 2026 

## **Union Types & Enums** 

**Union & Literal Types** 

Union: string | number — value can be either type. Literal: 'admin' | 'provider' | 'client' — only exact string values. Perfect for role types and narrowing logic in conditionals. 

**Enums & Status Types** Appointment status as an enum: PENDING | APPROVED | REJECTED CANCELLED | COMPLETED Enums group related constants with readable names. Use union literals for simple cases, enums when labels are shared across the codebase. 

May 2026 

## **TypeScript in Real Projects** 

**Reading TypeScript Errors** Errors point to exact lines. Read from the bottom up. Common messages: 'Property does not exist on type' and 'Type X is not assignable to type Y'. They protect your project from runtime bugs before you run a single line. 

**Where You Will Use It** Type API responses and request bodies. Type React component props. Type form data inputs. Share types between frontend and backend — one source of truth for all your data shapes. 

May 2026

## **Who Are You?** 

## **Authentication Explained** 

Authentication answers: who is this user? It verifies identity before granting access. Your appointment system uses it for login, registration, and all protected routes. 

Password hashing — never store plain text. Use bcrypt to hash before saving. Session-based — server stores session state. 

Token-based — server issues a token the client stores and sends back with every request. 

May 2026 

## **Lesson 2 JWT & Token Auth** 

## **How JWT Works** 

A JWT has three parts: header, payload, signature. The server signs it — only the server can verify it. Access token — short-lived. Refresh token — long-lived, used to get a new access token. 

Send in the Authorization header: Bearer <token>. Protected routes verify the token before processing. Prefer httpOnly cookies over localStorage in production. 

May 2026 

## **Lesson 3 Authorisation & RBAC** 

**Authentication vs Authorisation Role-Based Access Control** 

Authentication = who you are. Authorisation = what you are allowed to do. A user can be authenticated but still unauthorised to access certain resources or routes. Admin — full access to users, services, and appointments. Provider — manage assigned appointments only. Client — request appointments and view own history. Middleware enforces role checks on every protected route. 

May 2026 

## **Lesson 4 Secure Backend Practices** 

## **Protecting the API** 

Store secrets in environment variables — never hardcode credentials. Validate all incoming input before processing. 

Apply rate limiting to prevent brute force attacks. 

Use helmet.js to set secure HTTP response headers. 

**Passwords & API Security** Always hash passwords with bcrypt before storing. Enforce strong password policies. Verify API tokens and keys on every protected request. 

Never expose sensitive data in error messages or server logs. 

Log all security-relevant events.

# **Lesson 1 Introduction to Databases** 

May 2026 

**Relational vs Non-Relational What Is a Database?** A database stores and organises data so it can be retrieved, updated, and managed. Your appointment system uses one to store users, services, appointments, and status history. 

Relational (SQL) — fixed schema, tables, powerful joins, strong consistency. Non-relational (NoSQL) — flexible schema, horizontal scaling, suited for varied or fast-changing data structures. 

May 2026 

# **Lesson 2 SQL & NoSQL Database Types** 

## **Types to Know** 

SQL — tables with rows, columns, and a defined schema. Foreign keys link related tables. The appointments table references users and services by ID. 

## **NoSQL types:** 

Document — JSON-like records (MongoDB). Key-value — fast lookups by key (Redis). Column-family — wide-column storage (Cassandra). Graph — relationship-focused data (Neo4j). 

May 2026 

## **Lesson 3 MongoDB Fundamentals** 

**Core Concepts** 

Collections hold documents. Documents are BSON/JSON-structured records. Embed related data inside a document or reference it across collections using IDs. 

**Querying Documents** 

Query using filter objects: { status: 'PENDING' }. Filter by field value, range, or nested property. MongoDB's flexible schema allows fast iteration on evolving data shapes. 

May 2026 

## **Lesson 4 PostgreSQL Walkthrough** 

**What Is PostgreSQL?** 

PostgreSQL is an open-source relational database. Data lives in tables with typed columns and a defined schema. Every row is one record. The appointment system uses PostgreSQL to store users, roles, services, appointments, and status history. 

## **Core SQL Operations** 

SELECT — read records from a table. INSERT — add a new record. UPDATE — modify an existing record. DELETE — remove a record. JOIN — combine data from related tables. Foreign keys link appointments to their users and services. 

May 2026 

## **Key Questions to Ask** 

# **Choosing the Right Database** 

What does the data look like? How will it be queried? Does it need strict consistency or flexible structure? 

**For the Appointment System** 

Use PostgreSQL for users, roles, services, and appointments — structured relational data with clear foreign keys and strong consistency. Avoid NoSQL when your data has fixed relationships and predictable query patterns.

## **From Browser to Server** 

**The Request Lifecycle** → Browser looks up the domain via DNS resolves → to an IP address connects on a port (80 for → server receives the HTTP, 443 for HTTPS) request and sends back a response. 

DNS — translates domain names to IP addresses. Hosting — where your server lives (cloud, VPS, or dedicated). Ports — determine which service handles the request. HTTP is unencrypted. HTTPS encrypts traffic using TLS. 

May 2026 

## **Lesson 2 Web Servers** 

## **Nginx & Apache** 

A web server receives HTTP requests and sends back responses. It can serve static files directly or forward requests to your backend. Nginx — high-performance, lightweight, the standard choice for production. Apache — flexible and battle-tested, common in shared hosting. Both can act as reverse proxies, routing traffic to your Node.js backend. 

May 2026 

## **Lesson 3 Reverse Proxy & Load Balancing** 

**Reverse Proxy** 

**Load Balancing** 

A reverse proxy sits in front of your backend. Nginx receives all traffic, handles HTTPS termination, then forwards plain HTTP to your Node.js app running internally on a local port. Distributes traffic across multiple instances to prevent overload. Also enables routing multiple apps from one server using different domains or URL paths. 

May 2026 

## **Lesson 4 SSL/TLS & HTTPS Why HTTPS Matters** 

HTTPS encrypts data between browser and server using TLS. Protects credentials, tokens, and personal data in transit. Required by modern browsers and impacts search ranking. All HTTP connections should redirect to HTTPS. 

## **Certificates & Let's Encrypt** 

An SSL certificate verifies your domain and enables encryption. Let's Encrypt provides free, automated certificates. Use Certbot to issue and auto-renew on your server. No excuse for running HTTP in production. 

May 2026 

## **Lesson 5 Deployment Environments** 

**The Four Environments** 

Local — your machine. Development — shared dev server. Staging — mirrors production, used for final testing. Production — live, real users. Each has its own environment variables and config. 

**Deployment Checklist** 

Environment variables set. HTTPS enabled. Error handling active. Database connected. Logging in place. Health check responding. README updated.

## **Containers vs Virtual Machines** 

Docker packages your app and everything it needs — code, runtime, dependencies, config — into a container. Containers are lightweight, portable, and run the same everywhere. 

Image — the blueprint. Built from a Dockerfile. Container — a running instance of an image. Docker Hub — a registry to share and pull images. Containers share the OS kernel. VMs run a full OS.

## **Lesson 2 Docker Compose** 

**Multi-Container Applications** 

Docker Compose defines and runs multi-container apps from one YAML file. Define your backend, database, and frontend containers together. Volumes — persist data between container restarts. Networks — containers talk to each other by service name. For the appointment system: one compose file spins up Node.js, PostgreSQL, and Redis together.
